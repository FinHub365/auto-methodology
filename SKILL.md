# Auto方法论：自适应自主优化工具生成器

## 这个技能是什么

这是一个**元技能**——它不直接帮你优化某个具体的东西，而是帮你**设计和构建**一套完整的自主优化系统。

它的核心思想来自Andrej Karpathy的AutoResearch项目。Karpathy发现了一个惊人的简洁模式：只需要一个630行的Python脚本和一个Markdown指令文件，AI agent就能在一夜之间自主运行100多次实验，发现连他这样的世界级ML研究者都遗漏的优化。Shopify用同样的模式让模板引擎渲染速度提升了53%，内存分配减少了61%。

这个模式之所以强大，是因为它抓住了自主优化的三个本质要素——Karpathy称之为**三原件**（Three Primitives）：

1. **一个可编辑的资产**——agent能修改的那个文件
2. **一个标量指标**——一个数字，告诉你是变好了还是变差了
3. **一个时间限定的循环**——固定时长，让每次实验公平可比

只要你的问题能被映射到这三个要素上，这个模式就能帮你。无论是训练神经网络、优化SQL查询、调整系统配置、改进文案质量，还是提升API响应速度。

---

## 核心原则

在进入具体流程之前，先内化这些原则——它们贯穿整个方法论：

- 把目标压缩到一句话，只有一个核心结果。
- 选一个北极星指标。最多附加两个软约束。
- 冻结大部分世界。每次循环只改一个明确的可变面。
- 先建立基线，再尝试改进。
- 让每次迭代足够便宜，才能跑足够多次。
- 记录每一次运行，包括失败和崩溃。
- 只在测量到的胜利上前进。回退或丢弃退步。
- 更简洁的胜出方案优于略好但脆弱的复杂方案。
- 在破坏性操作、生产环境操作、或任何离开本机的操作之前，先征求确认。

## 使用此技能的完整流程

### 第一步：理解用户的领域和目标

在做任何事之前，先搞清楚三个问题：

**问题一：你想优化什么？**
不要接受模糊的回答。"让我的网站更快"不够好，"把首屏加载时间从3.2秒降到2秒以下"才是可以工作的目标。帮用户把模糊的愿望变成具体的、可测量的目标。

**问题二：你现在的基线是什么？**
没有基线就没有进步可言。帮用户建立当前状态的测量——跑一次基准测试、记录当前指标、拿到一个"实验零号"的结果。

**问题三：什么算"够好了"？**
帮用户设定一个合理的停止条件。不是所有优化都需要跑到极限——有时候10%的提升就已经满足业务需求了。但也要提醒用户，AutoResearch模式的魅力在于：很多看似微小的改进可以叠加成显著的提升。Karpathy的700次实验中只有约20次产生了改进，但这20次叠加起来让训练时间缩短了11%。

### 第二步：识别三原件

这是整个方法论的核心环节。每一个成功的Auto工具都建立在正确识别这三个要素之上。

#### 原件一：可编辑资产（The Editable Asset）

这是agent唯一被允许修改的文件。选择它时要遵循几个原则：

**单一修改点原则**：只给agent一个文件去修改。这不是限制，而是力量——它让每次实验的变化可追踪、可回滚、可理解。如果agent需要同时修改多个文件，那通常意味着你的问题分解还不够细。

**可解释性原则**：这个文件应该是人类可读的。当agent做出一个改进时，你应该能打开文件，理解它改了什么。这就是为什么AutoResearch用一个630行的Python文件，而不是一个二进制模型权重文件。

**自包含原则**：这个文件应该包含所有agent需要实验的"旋钮"。在AutoResearch中，train.py包含了模型架构、优化器配置、超参数——所有可以调整的东西都在一个地方。

帮用户找到这个文件。一些领域的映射关系：

| 领域 | 可编辑资产 | 里面包含什么 |
|------|-----------|-------------|
| ML训练 | train.py | 模型架构、优化器、超参数 |
| Web性能 | 目标页面源码或配置文件 | 渲染逻辑、资源加载策略、缓存配置 |
| SQL优化 | queries.sql | 查询语句、索引策略、执行计划提示 |
| 提示词工程 | prompts.md | 系统提示词、Few-shot示例、输出格式指令 |
| 编译优化 | build config文件 | 编译选项、链接策略、并行度配置 |
| 内容创作 | draft.md | 文案、结构、标题、CTA |
| API性能 | handler代码文件 | 路由逻辑、中间件、序列化策略 |
| 系统运维 | config文件 | 系统参数、资源分配、调度策略 |

如果用户的领域不在列表中，用以下思考框架帮助识别：
- "如果你要手动做一次优化尝试，你会去改哪个文件？"
- "你的同事们在优化这个问题时，通常调整的是什么？"
- "有没有一个文件，改了它之后可以直接看到效果？"

#### 原件二：标量指标（The Scalar Metric）

这是一个单一的数字——**越低越好**或**越高越好**，没有歧义。agent用它来做唯一的判断："这次改动是否让事情变好了？"

好的标量指标有几个特征：

**客观可计算**：不能依赖人类判断。"这段文案读起来更流畅了"不是标量指标，但"Flesch可读性指数从45提升到62"是。

**对操纵免疫**：指标的计算方式应该和agent修改的文件分离。这是AutoResearch最重要的设计之一——评估函数锁在不可变的prepare.py里，agent无法通过改写评分逻辑来"作弊"。如果agent既能改代码又能改评分方式，它一定会找到捷径。

**对词汇表/配置无关**：在AutoResearch中，val_bpb（每字节比特数）是词汇表大小无关的，这意味着agent可以自由实验不同的分词策略而不会导致不公平的比较。选指标时问一问："如果agent改变了某个底层配置，这个指标的比较还公平吗？"

**快速可得**：指标应该在每个实验周期内就能算出来。如果需要跑一天才能得到结果，你的循环就太慢了。

帮用户找到或构造这个指标：

| 领域 | 推荐指标 | 方向 | 为什么选它 |
|------|---------|------|-----------|
| ML训练 | val_bpb / val_loss | 越低越好 | 直接反映模型质量，快速可算 |
| Web性能 | Lighthouse Performance分数 | 越高越好 | 综合且标准化 |
| SQL优化 | 平均查询执行时间(ms) | 越低越好 | 直接且可重复测量 |
| 提示词工程 | 在评估集上的准确率 | 越高越好 | 客观可验证 |
| 编译优化 | 完整构建时间(秒) | 越低越好 | 直接且无歧义 |
| 内容创作 | 可读性指数 + 关键词密度复合分 | 越高越好 | 可量化的质量代理 |
| API性能 | P95延迟(ms) | 越低越好 | 比平均值更能反映真实体验 |
| 系统运维 | 目标SLA指标 | 视情况 | 直接对齐业务目标 |

**复合指标的处理**：有时用户需要同时优化多个维度。AutoResearch的做法是设一个"主指标"和"软约束"——主指标用于决定保留/回滚，软约束只是提醒agent不要走极端。例如：主指标是val_bpb，软约束是"显存不要爆炸性增长"。帮用户用同样的模式处理多维目标。

#### 原件三：时间限定循环（The Time-boxed Cycle）

每次实验的时长是固定的。这个设计看似简单，实则精妙——它确保了公平比较。在AutoResearch中，每次实验固定5分钟壁钟时间。一个在5分钟内达到更低val_bpb的模型就是更好的模型，无论它有多少参数、用了多大batch size。

帮用户确定合适的循环时长：

**太短**：实验来不及展现效果，噪声太大，agent会倾向于只做表面改动。
**太长**：循环次数太少，一夜只能跑几个实验，探索空间太有限。
**刚好**：足够看到有意义的差异，又短到能在一个工作日或一夜内跑几十次。

经验法则：
- 如果你的目标任务本身跑一次需要T秒，那么循环时长设为略大于T
- 如果T本身就很短（比如<10秒），可以在一个循环内跑多次取平均，减少噪声
- 如果T很长（比如>30分钟），考虑能否用一个更小的代理任务来替代——比如用子数据集代替全量数据

### 第三步：选择Auto原型

识别三原件后，为你的问题选择一个**主原型**。原型决定了循环内部的思维模式和操作节奏。先跑通一个原型，等第一个循环稳定后再考虑组合。

| 原型 | 循环模式 | 适用场景 |
|------|---------|---------|
| **AutoResearch** | 提出假设 → 跑实验 → 测量 → 保留或丢弃 | ML训练、科学计算、算法探索 |
| **AutoBuild** | 改代码/配置 → 跑测试/基准 → 保留或回滚 | 性能优化、编译优化、代码重构 |
| **AutoSearch** | 搜索 → 过滤 → 排序 → 综合 → 输出带置信度的结果 | 信息检索、文献综述、竞品分析 |
| **AutoOperate** | 检查状态 → 分类 → 执行动作 → 验证 → 记录 | 运维自动化、监控响应、流程编排 |
| **AutoContent** | 生成草稿 → 用评分标准打分 → 修订 → 只保留更高分的变体 | 文案优化、提示词工程、内容迭代 |
| **AutoAnalysis** | 生成候选解释/决策 → 用证据检验 → 提升最强的那个 | 数据分析、根因分析、商业决策 |

**选择指南**：
- 如果你的问题有明确的代码/配置可以改、有测试可以跑 → **AutoBuild**
- 如果你在探索未知空间、需要形成假设再验证 → **AutoResearch**
- 如果核心是"生成内容并评判质量" → **AutoContent**
- 如果核心是"从大量信息中提炼可信结论" → **AutoSearch** 或 **AutoAnalysis**
- 如果核心是"监控状态并自动响应" → **AutoOperate**
- 如果领域不明确，从 **AutoAnalysis** 或 **AutoResearch** 开始——它们能帮你先摸清问题结构，再转向更专门的原型

原型不是死板的分类，而是帮agent建立正确的操作心智模型。在program.md中明确告诉agent它属于哪个原型，以及这个原型的循环节奏是什么。

### 第四步：构建四层架构

AutoResearch的成功在于职责的严格分离。整个系统由四个**面**（Surface）组成，每个面有明确的权限边界：

| 面 | 职责 | 谁控制 | 可变性 |
|----|------|--------|--------|
| **纲领面** (Program Surface) | 目标、约束、行为规则、晋升规则 | 人类 | 人类手动修改 |
| **可变面** (Mutable Surface) | agent唯一被允许修改的资产 | Agent | 每次循环可变 |
| **评判面** (Oracle Surface) | 评估函数、测试套件、评分标准 | 基础设施 | 不可变 |
| **账本面** (Ledger Surface) | 运行日志、决策记录、经验教训 | Agent追加 | 只追加不修改 |

**关键规则：不要让一次实验同时改变可变面和评判面。** 否则循环失去可比性——agent可以通过降低评分标准来"伪造"进步。

#### 纲领面：program.md

这是你写给agent的"研究纲领"——它定义了方向、约束和行为规则。agent只读不写。

一个好的program.md包含以下几个部分：

**目标声明**：清晰地告诉agent它在优化什么。不只是指标，还有上下文和意义。"你正在优化一个语言模型的训练效率，目标是在固定时间内达到尽可能低的验证集bits-per-byte"比"降低val_bpb"要好得多。

**搜索空间描述**：告诉agent它可以尝试什么。在AutoResearch中，这包括"架构变化、超参数调整、优化器配置——一切都是公平的"。同时也说清楚什么不能动——"不要修改数据加载逻辑"。

**约束条件**：
- 硬约束：违反就必须回滚。例如"所有单元测试必须通过"。
- 软约束：可以适度违反，但agent应该意识到代价。例如"显存使用不应急剧增长"。

**简洁性准则**：这一条直接来自Karpathy的program.md——"在其他条件相同的情况下，更简洁的方案更好。一个引入丑陋复杂性的小改进不值得保留。"这条规则在agent跑了几百次实验之后变得尤为重要，因为它防止了代码腐烂。

**行为指令**：
- "永不停止"——agent应该无限循环直到被人类手动终止
- "不要暂停来问人类"——自主决策，不要中断循环
- "遇到崩溃时，读取日志，尝试修复简单bug，如果无法修复就记录并继续下一个实验"

生成program.md时，请阅读 `references/program-template.md` 获取完整模板。

#### 可变面：可编辑资产

就是第二步中识别的那个文件。确保它：
- 有清晰的初始状态（已经是可运行的基线版本）
- 有足够的注释让agent理解各个部分的作用
- 没有无关的代码——agent的修改空间越干净越好

#### 评判面：不可变评估

这一层包含agent不能修改的一切——数据加载、评估函数、测试套件。它的设计原则只有一条：**agent无法通过修改评估方式来伪造进步**。

具体需要创建的：

**评估脚本**（evaluate.py / evaluate.sh / 等价物）：
- 接收agent的修改产物（训练好的模型、编译好的程序、生成的内容）
- 在标准化条件下计算标量指标
- 输出格式应该简单明确，便于agent用grep或正则提取。推荐格式：`metric_name: value`

**数据/测试集**：
- 如果是ML场景，固定的验证集
- 如果是性能优化，固定的benchmark输入
- 如果是内容生成，固定的评估标准或参考集

**依赖锁定**：
- 固定所有依赖版本，避免环境变化引入噪声

#### 账本面：实验日志

accounts.tsv（或等价物）是只追加的结构化日志。详见第五步的棘轮机制。

### 第五步：设置棘轮机制与晋升规则

棘轮是保证单调进步的关键。整个机制用Git实现，极其简洁：

```
循环开始
├── agent读取program.md和历史实验记录
├── agent提出假设并修改可编辑资产
├── git commit（描述本次修改）
├── 运行实验（固定时长）
├── 提取指标
├── 晋升判断（见下方规则）
│   ├── 晋升 → 保留commit（新基线）✓
│   ├── 丢弃 → git reset --hard HEAD~1（回滚）✗
│   └── 崩溃 → 读取日志，记录错误，回滚 ✗
├── 将结果追加到 results.tsv
└── 回到循环开始
```

#### 晋升规则（Promotion Rules）

一个改动被晋升（保留为新基线）**当且仅当**满足以下条件之一：

1. **主指标改善**，且不违反任何硬约束
2. **主指标持平**，但方案**显著更简洁**（更少代码、更少依赖、更容易理解）
3. **用户在program.md中明确声明的权衡**——例如"愿意用5%的精度换取50%的速度提升"

一个改动必须被**丢弃或回滚**，当：

- 主指标退步
- 违反任何硬约束
- 主指标的"改善"来自于改变了评分方式而非真正的进步
- 胜出幅度极小但引入了大量复杂性

**不确定时不晋升**。如果证据模糊（比如改善在噪声范围内），将候选标记为 `hold`，继续下一个实验。

#### results.tsv的格式

```
commit	metric	resource	status	description
a1b2c3d	0.987	4.2GB	keep	增大注意力头数从8到12
e4f5g6h	0.991	4.1GB	discard	尝试余弦学习率但效果更差
i7j8k9l	N/A	N/A	crash	激活函数修改导致维度不匹配
m0n1o2p	0.987	3.8GB	keep	简化注意力实现，指标持平但代码更简洁
q3r4s5t	0.986	4.0GB	hold	改善在噪声范围内，待验证
```

这个日志文件是agent的"实验记忆"。它在每个循环开始时读取，用来：
- 避免重复已经失败的实验
- 识别有前途的探索方向
- 估计不同类型修改的成功率

### 第六步：生成完整的Auto工具

到这一步，你已经帮用户确定了三原件、选择了原型、设计了四层架构、建立了棘轮机制。现在把这些组装成一个完整的、可以交给agent运行的工具。

生成以下文件：

1. **Auto Brief** — 一份紧凑的自然语言摘要，包含：
   - goal: 系统要改进什么
   - metric: 如何衡量成功
   - archetype: 选定的Auto原型
   - budget: 每个循环允许的时间/tokens/算力/人工审查量
   - mutable surface: agent可以改什么
   - frozen surface: agent不能改什么
   - oracle: 用于评分的评估器/测试/标准
   - promotion rule: 什么时候保留一个改动
   - stop rule: 什么时候暂停、升级或终止循环
   - risk class: safe / caution / high-risk
2. **program.md** — 基于模板（见references/program-template.md），填入用户的具体领域信息
3. **可编辑资产文件** — 如果用户已有文件，帮助清理和准备；如果需要从头创建，生成一个可运行的基线版本
4. **评估脚本** — 实现不可变的指标计算逻辑
5. **run_loop.sh**（或等价物）— 包装棘轮循环的启动脚本，包含超时控制和日志记录
6. **results.tsv** — 空的实验记录表，带表头

### 第七步：启动前检查

在真正启动自主循环之前，和用户确认以下清单：

- [ ] 基线实验能成功运行并产出指标值吗？（跑一次验证）
- [ ] 评估脚本是否正确提取了指标？（检查输出格式）
- [ ] Git仓库已初始化？初始状态已提交？
- [ ] program.md中的约束条件是否完整？有没有遗漏的"绝对不能动"的部分？
- [ ] Auto Brief中的晋升规则和停止规则是否清晰？
- [ ] 循环时长设置合理吗？（估算一夜能跑多少实验）
- [ ] 资源约束是否明确？（磁盘空间、GPU显存、API调用限额等）
- [ ] 风险等级是否匹配？（safe可以全自主，caution需要阶段检查，high-risk需要逐步审批）

### 第八步：启动与监控

启动agent时，使用类似这样的提示词：

> "请阅读 program.md，了解你的任务和约束。然后检查 results.tsv 查看之前的实验记录。开始你的下一个实验——修改 [可编辑资产]，运行实验，评估结果，根据晋升规则决定保留或回滚。将每次实验记录到 results.tsv。永不停止，持续循环直到我手动终止你。"

**监控建议**：
- 定期检查results.tsv，观察指标趋势
- 如果连续多次实验都被discard，可能agent陷入了局部最优——考虑在program.md中给出新的探索方向
- 如果频繁crash，检查agent的修改是否超出了可编辑资产的合理范围
- 大约50-100次实验后是一个好的检查点——review agent的发现，决定是否调整方向

## 方法论护栏

以下五条是**不可违反的方法论禁令**。它们不是建议，而是整个Auto模式能可靠运转的前提条件：

### 1. 不可测则不优化

如果你无法用一个稳定的、自动化的评判面来量化结果，就不要启动自主循环。"感觉变好了"不算。先建立测量，再谈优化。

### 2. 不可事后改定义

agent不能在看到结果之后重新定义什么叫"成功"。指标、晋升规则、评分标准必须在循环开始前锁定。如果需要调整，暂停循环、人类修改、重新开始——而不是让agent在运行中悄悄改规则。

### 3. 循环不稳定时不扩范围

如果当前的单循环还会崩溃、指标还不稳定、基线还没跑通——不要急着加功能、加原型组合、加并行化。先让最小的循环可靠运转。**如果agent卡住了，缩小范围而不是增加更多活动部件。**

### 4. 更多自主 ≠ 更好方法论

自主运行是手段不是目的。一个需要人类每50次检查一次的循环，比一个完全自主但方向错误的循环有价值得多。根据风险等级设置合适的人类检查点。

### 5. 单循环跑通前不加编排层

不要在第一个循环还没产出可靠结果之前，就开始设计多agent编排、流水线、调度系统。先证明一个agent、一个循环、一个指标能产出价值，再考虑扩展。

## 领域适配的深层思考

### 不是所有问题都适合Auto模式

在帮用户构建Auto工具之前，先评估这个问题是否真的适合自主循环优化。适合的问题有以下特征：

**解空间巨大但每次评估成本低**：超参数调优、配置优化、提示词迭代——这些问题有无数种可能的组合，但验证每一种只需要很短的时间。

**微小改进可以叠加**：优化编译速度、降低API延迟、提升模型精度——每次1%的改进看似微不足道，但50次这样的改进叠加起来效果惊人。

**当前方案已经"差不多好了"但还有提升空间**：如果当前方案根本不能工作，需要的是重新设计而不是迭代优化。Auto模式最适合在一个已经可用的基线上精雕细琢。

**不适合的问题**：
- 需要人类创意判断的（设计风格、品牌调性）
- 评估成本极高的（需要真实用户反馈、需要跑一周的A/B测试）
- 解空间极小的（只有两三种可能的配置）
- 风险极高的（生产环境的核心基础设施——不要让agent自主修改你的数据库schema）

### 领域默认映射

当用户的领域明确时，使用以下默认配置作为起点：

- **编码工作**：冻结API契约和测试套件，只允许修改目标模块，用测试通过率+一个基准指标评分
- **研究工作**：冻结纳入规则和输出模式，允许修改搜索策略和综合方法，用来源质量、时效性和置信度评分
- **运维工作**：冻结安全和审批边界，允许修改调度、路由或运行手册，用解决率、延迟和事件数评分
- **内容工作**：冻结品牌和政策规则，允许修改提示词、结构和示例，用显式评分标准或下游响应指标评分
- **分析工作**：冻结定义和数据溯源，允许修改框架、分群和假设顺序，用正确性、实用性和决策清晰度评分

### 并行化：从单GPU到GPU集群的思维

Karpathy的原始模式是串行的——一个agent，一个GPU，一次一个实验。但SkyPilot团队展示了如何把它扩展到16个GPU并行运行，8小时跑了910个实验。

如果用户有多机资源，帮他们思考并行化策略：
- **独立并行**：多个agent各自独立运行，最后合并最佳结果
- **共享进展**：agent之间共享results.tsv，避免重复实验
- **分层策略**：不同agent探索不同的方向（如一个做架构变化，一个做超参数搜索）

### 何时停止

Auto循环理论上可以永远运行，但实践中应该在以下情况停止：
- 连续N次实验（比如20次）都没有产生改进——可能已经接近局部最优
- 已经达到用户设定的目标
- 资源预算耗尽
- 代码复杂度开始失控（违反简洁性准则）

## 核心理念总结

Karpathy用一句话概括了Auto方法论的本质："你不再直接编写Python文件了。你在编写program.md——你在编程那个程序。"

这揭示了一个深刻的范式转变：

- **传统方式**：人类写代码 → 人类测试 → 人类改进 → 循环
- **Vibe Coding**：人类描述需求 → AI写代码 → 人类审查 → 循环
- **Auto方法论**：人类写研究纲领 → AI自主实验 → AI自主评估 → AI自主改进 → 人类偶尔检查方向

人类的角色从"实现者"变成了"方向制定者"。你不需要知道最佳的学习率是多少，你只需要知道"学习率是一个值得探索的方向"。Agent会帮你找到那个最佳值——以及你可能根本没想到去调整的其他东西。

这就是为什么这个方法论是一个**元技能**——它不告诉你优化什么，它教你**如何构建一套系统，让AI自己去发现怎么优化**。
