## 1. ML模型训练（AutoResearch原版）

这是Karpathy的原始应用场景，作为其他领域的参照。

**三原件：**
| 原件 | 具体实例 |
|------|---------|
| 可编辑资产 | `train.py`（630行，含模型架构、优化器、超参数） |
| 标量指标 | `val_bpb`（验证集bits-per-byte，越低越好） |
| 循环时长 | 5分钟壁钟时间 |

**不可变层：**
- `prepare.py`：BPE分词器训练、数据加载、`evaluate_bpb()`函数
- 训练/验证数据集分割

**实际成果：**
- 700次实验，约20次改进
- 发现QK-Norm缺失乘法器、V-Embedding正则化价值、注意力带宽过于保守等问题
- 训练时间从2.02小时降至1.80小时（11%提升）

**关键洞察：** 每次实验固定壁钟时间而非固定步数，使得agent可以自由改变模型大小和batch size，系统自然选择在给定硬件上最高效的配置。

## 2. Web前端性能

**场景：** 电商网站首页加载过慢，需要优化Core Web Vitals。

**三原件：**
| 原件 | 具体实例 |
|------|---------|
| 可编辑资产 | `index.html` + 内联样式和脚本（或打包后的单一入口文件） |
| 标量指标 | Lighthouse Performance分数（0-100，越高越好） |
| 循环时长 | 2分钟（构建 + Lighthouse CI运行） |

**不可变层：**
- Lighthouse CI配置和运行脚本
- 测试用的网络条件配置（模拟3G/4G）
- 页面功能完整性测试（确保优化不破坏功能）

**program.md中的约束：**
- 硬约束：页面所有功能必须正常工作（由端到端测试验证）
- 硬约束：页面视觉外观不能改变（由截图比对验证）
- 软约束：JavaScript bundle大小不应超过初始值的150%

**评估脚本示例：**
```bash
#!/bin/bash
npm run build
lighthouse http://localhost:3000 --output json --quiet | jq '.categories.performance.score * 100'
echo "lighthouse_score: $score"
```

**探索方向：**
- 图片懒加载策略和格式转换（WebP/AVIF）
- 关键CSS内联，非关键CSS延迟加载
- JavaScript代码分割和动态导入
- 字体加载策略（font-display: swap, preload）
- 资源预加载提示（preconnect, prefetch）
- 服务端组件/静态生成比例调整
- 第三方脚本加载时机优化

## 3. SQL查询优化

**场景：** 数据分析平台的核心报表查询需要30秒，用户体验极差。

**三原件：**
| 原件 | 具体实例 |
|------|---------|
| 可编辑资产 | `queries.sql`（包含查询语句、CTE结构、索引建议注释） |
| 标量指标 | 查询执行时间的P95（毫秒，越低越好） |
| 循环时长 | 1分钟（执行查询5次取P95） |

**不可变层：**
- 数据库实例和测试数据（固定的数据快照）
- 执行和计时脚本
- 结果正确性校验（查询结果hash必须与基线一致）

**program.md中的约束：**
- 硬约束：查询结果必须与基线完全一致（hash校验）
- 硬约束：不能修改表结构（只能建议索引，但索引创建算在时间预算内）
- 软约束：避免创建过多索引影响写入性能

**评估脚本核心逻辑：**
```bash
# 跑5次取P95
for i in {1..5}; do
  time_ms=$(psql -c "\timing on" -c "$(cat queries.sql)" 2>&1 | grep "Time:" | awk '{print $2}')
  times+=($time_ms)
done
# 排序取P95
p95=$(echo "${times[@]}" | tr ' ' '\n' | sort -n | tail -1)
echo "query_time_p95_ms: $p95"
# 结果校验
result_hash=$(psql -t -c "$(cat queries.sql)" | md5sum | awk '{print $1}')
echo "result_hash: $result_hash"
```

**探索方向：**
- CTE重构为子查询或反之
- JOIN顺序和类型调整
- 条件下推和谓词优化
- 索引策略（覆盖索引、部分索引、表达式索引）
- 窗口函数替代自关联
- 物化视图或预计算中间结果
- 查询执行计划提示（如有支持）

## 4. 提示词工程

**场景：** 构建一个从客户邮件中提取结构化信息的LLM管道，需要提升准确率。

**三原件：**
| 原件 | 具体实例 |
|------|---------|
| 可编辑资产 | `prompts.md`（系统提示词、Few-shot示例、输出格式指令） |
| 标量指标 | 在50条标注样本上的F1分数（越高越好） |
| 循环时长 | 3分钟（调用API评估全部样本） |

**不可变层：**
- 50条人工标注的测试样本（输入邮件 + 正确的结构化输出）
- 评估脚本（调用LLM API、解析输出、与标注比对、计算F1）
- LLM API配置（模型版本、temperature固定为0）

**program.md中的约束：**
- 硬约束：必须使用指定的模型和temperature
- 硬约束：输出格式必须是有效JSON
- 软约束：提示词长度不应超过2000 tokens（成本考虑）
- 软约束：API调用次数不应增加（不要通过多次调用+投票来提升准确率）

**特殊考虑：**
这个场景的"循环时长"主要受API速率限制和成本约束。如果评估集很大（比如500条），考虑用一个50条的子集做快速迭代，定期在全量集上验证。

**探索方向：**
- 系统提示词的措辞和角色设定
- Few-shot示例的选择和数量
- 输出格式（JSON schema vs 自由文本 vs XML）
- 思维链（Chain-of-Thought）策略
- 负面示例（展示常见错误和如何避免）
- 分步提取 vs 一次性提取
- 关键字段的特殊处理指令

## 5. 编译构建优化

**场景：** 大型C++项目完整构建需要45分钟，开发效率极低。

**三原件：**
| 原件 | 具体实例 |
|------|---------|
| 可编辑资产 | `CMakeLists.txt` 或 `Makefile`（编译选项、链接策略、目标定义） |
| 标量指标 | 完整构建时间（秒，越低越好） |
| 循环时长 | 60分钟（构建本身可能很长，所以循环也较长） |

**不可变层：**
- 源代码文件（不改代码，只改构建配置）
- 编译正确性验证（构建产物的功能测试）
- 计时脚本

**特殊考虑：**
- 循环时长较长意味着一夜只能跑十几个实验，需要agent更聪明地选择实验
- 可以用增量构建时间作为辅助指标
- 构建优化的很多改动是相互独立的（头文件预编译、并行度、链接方式），可以并行探索

**探索方向：**
- 预编译头文件（PCH）策略
- 并行编译度（-j参数）
- 链接器选择（ld vs gold vs lld vs mold）
- 编译器缓存（ccache/sccache配置）
- 模块化编译（C++20 modules）
- Unity builds / jumbo builds
- 优化级别和调试信息的权衡
- 依赖关系分析和不必要依赖的移除

## 6. 内容文案优化

**场景：** SaaS产品的落地页转化率不理想，需要优化文案。

**三原件：**
| 原件 | 具体实例 |
|------|---------|
| 可编辑资产 | `landing_copy.md`（标题、副标题、卖点、CTA、社会证明） |
| 标量指标 | 复合质量分 = 0.4×可读性 + 0.3×说服力 + 0.3×清晰度（LLM评分，1-100） |
| 循环时长 | 1分钟（LLM评分 + 可读性指数计算） |

**不可变层：**
- 评分提示词和评分LLM配置（固定的评估标准）
- 品牌语气指南（作为评分参考）
- 可读性计算脚本（Flesch-Kincaid等）
- 关键信息完整性检查（必须包含的产品特性列表）

**特殊考虑：**
这是一个"软"领域——标量指标是LLM评分而非客观测量。为了让这个方法有效：
- 评分提示词必须详细、具体、包含评分标准（rubric）
- 建议跑3次评分取平均，减少LLM评分的随机性
- 最终结果需要人类审查——数字好不代表文案真的好
- 设置硬约束确保必要信息不被删除

**探索方向：**
- 标题的不同公式（疑问句 vs 陈述句 vs 数据驱动）
- 卖点的排列顺序和优先级
- CTA的措辞和紧迫感
- 段落长度和节奏变化
- 社会证明的呈现方式
- 技术术语 vs 通俗表达的平衡
- 情感诉求 vs 理性诉求的比例

## 7. API服务性能

**场景：** 微服务的核心接口P99延迟过高，需要在不改变功能的前提下优化。

**三原件：**
| 原件 | 具体实例 |
|------|---------|
| 可编辑资产 | `handler.go`（请求处理逻辑、序列化、缓存策略） |
| 标量指标 | P99延迟（毫秒，越低越好） |
| 循环时长 | 3分钟（编译 + 30秒负载测试） |

**不可变层：**
- 负载测试脚本和配置（固定的请求模式和并发度）
- API契约测试（响应格式和正确性验证）
- 性能测量工具配置

**评估脚本核心逻辑：**
```bash
go build -o server . && ./server &
sleep 2
wrk -t4 -c100 -d30s --latency http://localhost:8080/api/target > bench.log
kill %1
p99=$(grep "99%" bench.log | awk '{print $2}')
echo "p99_latency_ms: $p99"
```

**探索方向：**
- 序列化库替换（encoding/json vs easyjson vs sonic）
- 内存分配优化（sync.Pool, 预分配buffer）
- 并发模型调整（goroutine池 vs 无限制）
- 缓存策略（本地缓存 vs Redis, 缓存粒度）
- 数据库查询批处理和连接池配置
- 中间件链优化（移除不必要的中间件）
- HTTP/2和连接复用配置

## 8. LLM推理优化

**场景：** 自部署的开源LLM推理服务吞吐量不足，需要在保持质量的前提下提升。

**三原件：**
| 原件 | 具体实例 |
|------|---------|
| 可编辑资产 | `serve_config.py`（量化配置、批处理策略、KV缓存、采样参数） |
| 标量指标 | 吞吐量 tokens/second（越高越好），质量门控：困惑度不超过基线的105% |
| 循环时长 | 5分钟（加载模型 + 跑benchmark prompt集） |

**不可变层：**
- 模型权重文件
- benchmark prompt集（100条标准化输入）
- 质量评估脚本（困惑度计算）
- 吞吐量测量工具

**特殊考虑：**
这个场景有一个典型的"质量门控"——吞吐量是主指标，但困惑度劣化超过5%的实验直接视为失败。这是AutoResearch"主指标+软约束"模式的变体，但这里的质量约束更适合作为硬约束。

**探索方向：**
- 量化级别（FP16 → INT8 → INT4，GPTQ vs AWQ vs GGUF）
- 批处理策略（连续批处理、动态批处理大小）
- KV缓存配置（PagedAttention参数、缓存大小）
- 张量并行度
- 投机解码（Speculative Decoding）参数
- Flash Attention配置
- 内存管理策略（预分配 vs 按需）

## 通用适配框架

如果用户的领域不在上述示例中，使用以下问题引导识别三原件：

### 找可编辑资产
1. "你平时优化这个问题时，打开的是哪个文件？"
2. "如果只能改一个文件来改善结果，你会改哪个？"
3. "这个文件是人类可读的吗？Agent能理解它吗？"
4. "改了这个文件后，能直接看到效果吗，还是需要一系列后续步骤？"

### 找标量指标
1. "你怎么判断'变好了'？用什么数字？"
2. "这个数字能自动算出来吗？需要多长时间？"
3. "如果agent改了底层配置，这个指标的比较还公平吗？"
4. "有没有一个简单的命令就能得到这个数字？"

### 定循环时长
1. "跑一次完整的评估需要多长时间？"
2. "你希望一夜能跑多少次实验？"
3. "有没有一个更小的代理任务可以快速给出近似结果？"
4. "在这个时长内，结果的方差大吗？需要多次取平均吗？"
